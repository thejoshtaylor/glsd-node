## Section Structure

### 1. Overview
- Protocol version (from `protocol.Version` constant: "1.2.0")
- Transport: WebSocket over TLS (`wss://`)
- Framing: JSON text frames, every message wrapped in an Envelope
- Direction convention: "inbound" = server-to-node, "outbound" = node-to-server

### 2. Envelope Format
- Document the `Envelope` struct: `type` (string), `id` (string), `payload` (raw JSON, omitempty)
- Explain `Payload` is `json.RawMessage` -- dispatch on `type` before decoding
- Document `Encode()` and `Decode()` functions
- Document `NewMsgID()`: 16 random bytes, hex-encoded (32 chars)
- Show a concrete JSON example of a wrapped message

### 3. Message Type Catalog
For EACH type constant in messages.go, create a subsection with:
- Type string (e.g., `"node_register"`)
- Direction (inbound or outbound; ACK is outbound)
- JSON schema showing every field with its Go type mapped to JSON type
- Field notes (e.g., `running_instances` must be `[]` not `null`; `session_id` is omitempty)
- When it is sent (trigger condition from the code)

Order: group by direction.

**Outbound (node-to-server):**
- `node_register` -- from `NodeRegister` struct. Sent on connect (first frame, before writer goroutine starts) and in response to `status_request`. Fields: node_id, platform, version, projects, running_instances. Note: RunningInstances has no omitempty -- always `[]` when empty.
- `ack` -- from `ACK` struct. Sent immediately after receiving `execute`, before spawning instance. Uses the inbound envelope's ID for correlation. Fields: instance_id.
- `stream_event` -- from `StreamEvent` struct. Sent per NDJSON line from Claude CLI. Fields: instance_id, data.
- `instance_started` -- from `InstanceStarted` struct. Sent when Claude CLI subprocess launches. Fields: instance_id, project, session_id (omitempty).
- `instance_finished` -- from `InstanceFinished` struct. Sent on clean exit. Fields: instance_id, exit_code.
- `instance_error` -- from `InstanceError` struct. Sent on error exit or rate limit rejection. Fields: instance_id, error.
- `node_disconnect` -- from `NodeDisconnect` struct. Sent before clean WebSocket close. Fields: reason (omitempty).

**Inbound (server-to-node):**
- `execute` -- from `ExecuteCmd` struct. Fields: instance_id, project, work_dir, prompt, session_id (omitempty).
- `kill` -- from `KillCmd` struct. Fields: instance_id.
- `status_request` -- from `StatusRequest` struct. Empty payload (no fields).

### 4. Authentication Handshake
- Document from `dial.go`: Bearer token in HTTP `Authorization` header during WebSocket upgrade
- Token source: `SERVER_TOKEN` env var via `NodeConfig.ServerToken`
- Server URL: `SERVER_URL` env var via `NodeConfig.ServerURL`
- First frame after connection: `node_register` (sent synchronously by `sendRegister` before writer/reader/heartbeat goroutines start)
- Server should validate token during HTTP upgrade (return 401/403 to reject)

### 5. Connection Lifecycle
- Dial: `websocket.Dial(ctx, serverURL, opts)` with Bearer header
- On success: send `node_register`, then start reader/writer/heartbeat goroutines
- On dial failure: exponential backoff with full jitter (AWS algorithm), range 500ms-30s, doubles each attempt, resets on successful connect
- On connection drop (read/write error): return to dial loop, apply backoff
- On clean shutdown: writer sends `node_disconnect` frame, then `conn.Close(StatusNormalClosure, "shutdown")`, then exits. No reconnect attempt after stop signal.

### 6. Heartbeat
- Interval: configurable via `HEARTBEAT_INTERVAL_SECS` (default 30s)
- Mechanism: `conn.Ping(ctx)` with timeout of 3x interval (default 90s)
- On pong timeout: cancel connection context, triggering reconnect
- Note: coder/websocket requires active reader goroutine for pong handling

### 7. Sequence Diagrams (Mermaid)

**Diagram 1: Connection establishment**
Node -> Server: WebSocket upgrade with Bearer token
Server -> Node: 101 Switching Protocols
Node -> Server: node_register (first frame)

**Diagram 2: Execute-stream-finish flow**
Server -> Node: execute {instance_id, project, prompt, ...}
Node -> Server: ack {instance_id} (correlation: uses execute envelope ID)
Node -> Server: instance_started {instance_id, project}
Node -> Server: stream_event {instance_id, data} (repeated, one per NDJSON line)
Node -> Server: instance_finished {instance_id, exit_code: 0}

**Diagram 3: Kill flow**
Server -> Node: kill {instance_id}
Node: cancels instance context
Node -> Server: instance_error {instance_id, error} OR instance_finished (whichever fires first via sync.Once gate)

**Diagram 4: Reconnect flow**
Node: detects connection drop
Node: backoff delay (jittered exponential)
Node -> Server: WebSocket upgrade with Bearer token
Node -> Server: node_register (with running_instances snapshot)

**Diagram 5: Clean shutdown**
Node: receives SIGINT/SIGTERM
Node: dispatcher stops accepting new commands
Node: waits for running instances to drain (10s timeout)
Node -> Server: node_disconnect {reason: "shutdown"}
Node: closes WebSocket (StatusNormalClosure)

### 8. Concurrency Model
- Single writer goroutine: ALL outbound frames go through `sendCh` (buffered 64) to writer goroutine. Only exception: `node_register` on connect (written directly before goroutines start).
- Reader goroutine: reads frames, decodes Envelope, forwards to `recvCh` (buffered 64, non-blocking send with drop on full).
- Heartbeat goroutine: periodic ping on its own ticker.
- Dispatcher goroutine: reads from `recvCh`, dispatches to handlers, spawns per-instance goroutines.

### 9. Error Handling
- Rate limited execute: immediate `instance_error` with error "rate limited" (no instance spawned)
- Unknown command type: logged and ignored
- Malformed envelope: logged and skipped
- Write failures: logged, connection will drop and reconnect
- recvCh full: inbound envelope dropped with warning log
