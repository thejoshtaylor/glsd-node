## Section Structure

### 1. Overview
- The server is the central management plane for GSD nodes
- Nodes connect outbound to the server (server does NOT connect to nodes)
- Communication: WebSocket over TLS, JSON text frames
- Reference: `docs/protocol-spec.md` for full wire protocol details
- Server responsibilities: accept node connections, dispatch commands to nodes, receive streaming output, track node/instance state

### 2. WebSocket Endpoint
- Endpoint: single WebSocket upgrade endpoint (e.g., `wss://server.example.com/ws/node`)
- Authentication: validate `Authorization: Bearer {token}` header during HTTP upgrade
  - Return 401 if token missing, 403 if token invalid
  - Token validation mechanism is server's choice (static secret, JWT, database lookup)
- After upgrade: expect `node_register` as the first frame from the node
  - If first frame is not `node_register`, close with error
- Connection parameters the server must handle:
  - Nodes reconnect automatically -- server must accept re-registration gracefully
  - Nodes send periodic WebSocket pings (default every 30s) -- server must respond with pongs (standard WebSocket behavior)
  - Nodes may disconnect and reconnect at any time -- `node_register` on reconnect includes `running_instances` snapshot for state reconciliation

### 3. Data Models

**Node model** (what the server must track per connected node):
- `node_id` (string): stable hardware-derived identifier, primary key for node identity. Same node reconnecting will have same node_id.
- `platform` (string): OS identifier (e.g., "windows", "linux", "darwin")
- `version` (string): node software version (currently "1.2.0")
- `projects` (string[]): project names the node has configured
- `connected_at` (timestamp): when the current WebSocket session started
- `last_heartbeat` (timestamp): last successful ping/pong exchange
- `status` (enum): connected, disconnected, stale

**Instance model** (what the server must track per running Claude CLI instance):
- `instance_id` (string): server-assigned UUID, unique across all nodes. Server generates this when dispatching an `execute` command.
- `node_id` (string): which node is running this instance
- `project` (string): project name
- `session_id` (string, optional): Claude session ID for resume capability. Populated from `instance_started` event. Empty until first response from Claude CLI.
- `status` (enum): pending (execute sent, no ACK yet), running (ACK or instance_started received), finished, errored
- `started_at` (timestamp): when instance_started was received
- `finished_at` (timestamp, optional): when terminal event received

**Relationship:**
- One node has zero or more instances
- Instance belongs to exactly one node
- On node reconnect: reconcile `running_instances` from `node_register` with server's tracked instances (remove stale, add unknown)

### 4. Command Dispatch

**Sending commands to a node:**

The server sends commands as Envelope-wrapped JSON frames. Each command uses a unique message ID (server should generate its own IDs, same format: random hex string).

**execute** -- Start a Claude CLI instance:
- Server generates `instance_id` (UUID) before sending
- Required fields: `instance_id`, `project`, `work_dir`, `prompt`
- Optional: `session_id` (to resume a previous Claude session)
- Expected response flow:
  1. `ack` (immediate, correlated by envelope ID) -- instance accepted
  2. `instance_started` -- subprocess launched
  3. `stream_event` (zero or more) -- NDJSON output chunks
  4. `instance_finished` OR `instance_error` -- terminal event, exactly one per instance
- If rate limited: immediate `instance_error` with error "rate limited" (no ACK, no instance_started)

**kill** -- Terminate a running instance:
- Required: `instance_id`
- Node cancels the instance's context; Claude CLI subprocess terminates
- Terminal event (`instance_finished` or `instance_error`) will follow -- server should wait for it rather than immediately marking as killed

**status_request** -- Query current node state:
- Empty payload
- Node responds with `node_register` containing current `running_instances`
- Uses the status_request's envelope ID for correlation

### 5. Inbound Event Handling

**Events the server receives from nodes:**

| Event | Trigger | Server Action |
|-------|---------|---------------|
| `node_register` | Connect, reconnect, status response | Upsert node record, reconcile instances |
| `ack` | After execute received | Mark instance as running, correlate via envelope ID |
| `stream_event` | Claude CLI output | Forward to frontend/store, keyed by instance_id |
| `instance_started` | Subprocess launched | Update instance status, capture session_id |
| `instance_finished` | Clean exit | Mark instance complete, record exit_code |
| `instance_error` | Error or rate limit | Mark instance errored, record error message |
| `node_disconnect` | Clean shutdown | Mark node disconnected, expect WebSocket close |

**Terminal event guarantee:** Each instance produces exactly one terminal event (`instance_finished` or `instance_error`). The node uses a `sync.Once` gate to prevent duplicates even in race conditions (e.g., kill + natural exit). Server can rely on this: no need to deduplicate terminal events.

### 6. State Reconciliation on Reconnect

When a node reconnects (sends `node_register` with `running_instances`):
1. Compare `running_instances` with server's tracked instances for that `node_id`
2. Instances in server but NOT in node's list: mark as lost/errored (node crashed during execution)
3. Instances in node's list but NOT in server: add to tracking (server restarted while node was running)
4. Instances in both: update session_id if changed, confirm running status

This is critical for reliability: neither node nor server crashes should cause permanent state divergence.

### 7. Node Health Monitoring

**Heartbeat-based health:**
- Nodes send WebSocket pings every 30s (configurable on node side)
- Server tracks `last_heartbeat` timestamp
- If no ping received for >90s, consider node stale
- If WebSocket connection drops, mark node disconnected immediately
- Stale/disconnected nodes: mark all their instances as lost

**Clean disconnect:**
- Node sends `node_disconnect` before closing WebSocket
- Server should mark node as disconnected (not crashed)
- All instances for that node should have already sent terminal events (node drains instances before disconnecting)

### 8. OpenAI Whisper Integration Point

**Purpose:** Server-side voice-to-text transcription for the web frontend. Users speak into the web UI; audio is sent to the server; server transcribes via OpenAI Whisper; transcribed text becomes the `prompt` field in an `execute` command.

**Integration architecture:**
- Frontend captures audio (browser MediaRecorder API)
- Frontend sends audio to server via REST endpoint (NOT via WebSocket -- binary audio does not belong in the JSON WebSocket protocol)
- Server calls OpenAI Whisper API (`POST https://api.openai.com/v1/audio/transcriptions`)
  - Model: `whisper-1`
  - Supported formats: mp3, mp4, mpeg, mpga, m4a, wav, webm
  - Max file size: 25MB
  - Returns: transcribed text
- Server uses transcribed text as the `prompt` in an `execute` command dispatched to the target node
- This is entirely server-side -- nodes have no knowledge of voice input

**Required server config:**
- `OPENAI_API_KEY` -- for Whisper API authentication
- Optional: language hint, response format preference

**Endpoint suggestion:**
- `POST /api/transcribe` -- accepts audio file, returns transcribed text
- Or integrate directly into the execute flow: `POST /api/execute` accepts either text prompt or audio file

### 9. Security Considerations

- **Token rotation:** Server should support rotating `SERVER_TOKEN` without dropping connected nodes (grace period for old token)
- **Node identity:** `node_id` is hardware-derived. Server should track first-seen timestamp and alert on new node_id values (potential unauthorized node)
- **Rate limiting:** Nodes enforce per-project rate limits locally; server may additionally enforce global rate limits
- **Command validation:** Server should validate `project` exists on the target node (check against `projects` list from `node_register`) before dispatching `execute`
- **Audit trail:** Log all commands dispatched and events received, with node_id and instance_id context

### 10. Deployment Topology

```
[Web Frontend] --REST/WS--> [Server] <--WSS-- [Node 1]
                                      <--WSS-- [Node 2]
                                      <--WSS-- [Node N]
                              |
                              v
                        [OpenAI Whisper API]
```

- Nodes initiate connections outbound (NAT-friendly, no firewall changes needed on node side)
- Server listens on a single port for both frontend and node WebSocket connections (different endpoints)
- Server is the only component that talks to OpenAI Whisper API
